**Status:** LIVE ‚Äî this document is never `FROZEN`; it is synchronized after every Specification Freeze session and on every platform-governance pass.
**Change class:** Project-runtime governance record. The register *mechanism* (schema, ID scheme, update protocol below) is reusable framework policy; the *entries* are project-specific (hotel-crm) evidence, consistent with Knowledge Separation (`../CLAUDE.md`).
**Scope:** Every unresolved issue surfaced by a completed Specification Freeze session (Documentation Workflow) across all module specifications in `docs/03-modules/`, plus cross-cutting/platform-level governance issues.

**Canonical framework version:** see [`../VERSION.yaml`](../VERSION.yaml).
**Access-boundary note (independent security review, framework 1.3.0):** this register consolidates, in one place, exact evidence for every live Critical/High security defect already independently documented in each module's own `MODULE_SPEC.md` and in `SYNC_STATE.yaml`; it discloses no new fact. This document's access control must not be looser than the strictest module specification it aggregates. Confirming and documenting the repository's/`.claude/`'s publication boundary (public vs. private, any mirroring or export) is a pending human decision, not resolved by this register.

## Purpose

Before this register existed, unresolved specification issues (open decisions, blocking findings, ownership gaps, migration gaps) were discoverable only by re-reading `SYNC_STATE.yaml` and every individual `MODULE_SPEC.md`'s own "Risks, Assumptions, and Open Decisions" section. This register is the single, permanent, human-scannable index of *all* such issues, across all modules, so that:

1. A future session (or human) can see the full unresolved-issue surface without re-reading every specification.
2. Issues are never silently rediscovered, duplicated, or dropped across sessions.
3. Resolution is tracked to `RESOLVED` with history preserved ‚Äî nothing is deleted.

This register **references** canonical sources (per [Context Artifacts ¬ß1.3 "Reference over copy"](../constitution/CONTEXT_ARTIFACTS.md)); it does not restate or supersede them. The canonical source for any issue is always the cited specification section or `SYNC_STATE.yaml` entry. Where this register and a canonical source disagree, the canonical source wins and the disagreement is itself logged as a new issue (Constitution ¬ß5).

## Schema

| Column | Definition |
|---|---|
| ID | Stable `SIR-<SCOPE>-<NNN>`, assigned once, never reused, never renumbered (Constitution ¬ß15; compound `<TYPE>-<SCOPE>-<NNN>` shape matches the `OQ-<MODULE>-<NNN>`/`FIND-<CLASS>-<MODULE>-<NNN>` convention already established across the module specifications this register aggregates) |
| Canonical source(s) | The `SYNC-*`, `OQ-*`, `FIND-*`, `OD-*`, or `ADR-*` ID(s) this entry aggregates, plus the spec path. A canonical source may be aggregated by more than one row when the rows are in different sections: a Global/Cross-Cutting row is the platform-wide statement, and a module-section row is a thin cross-reference back to it (e.g. every module's `SYNC-001` ownership row states "see `SIR-GLOB-001`") ‚Äî this is intentional per-module disposition tracking, not a duplicate; **within the same section, one canonical source ID gets exactly one row.** |
| Category | One or more of: `ownership` \| `architecture` \| `security` \| `decision-required` \| `migration-gap` \| `consistency` \| `knowledge-sync` \| `performance` ‚Äî comma-separated when an issue genuinely spans categories (e.g. `security, decision-required`) |
| Severity | Base value is one of `Critical` / `High` / `Medium` / `Low` / `Note`, per [Review Gates](../constitution/REVIEW_GATES.md) severity table (`decision-required` items with no live-code defect carry no severity ‚Äî marked `‚Äî`); may carry a parenthetical qualifier on top of the base value (e.g. `High (resolved in spec, verify implementation)`) when the base value alone would lose information the canonical source itself records ‚Äî the qualifier never substitutes for the base value |
| Summary | One-line statement of the unresolved issue |
| Evidence | Path-and-line, revision, or decision ID |
| Authority needed | `human` / `human/architecture` / `human/product` / `human/compliance` |
| Status | Base value is one of `OPEN` / `BLOCKED` / `RESOLVED`; may carry a free-text qualifier after an em-dash (e.g. `OPEN ‚Äî BLOCKING (Critical)`) mirroring the qualifier convention already used in every module specification's own Risks/Open-Decisions table ‚Äî the qualifier never substitutes for the base value |
| First recorded | Date/session this issue was first captured (in its canonical source or this register) |
| Last verified | Date this register entry was last checked against the repository |

`Unknowns` and `Limitations` are not separately tracked per entry here (that is the canonical source's job, per [Reviewer Findings](../constitution/REVIEWER_FINDINGS.md)); this register carries only status and cross-reference.

## Update Protocol (Workflow Integration)

This register is synchronized as a mandatory step of two existing workflows ‚Äî no new workflow is introduced:

1. **[Documentation Workflow](../workflows/documentation.md)** ‚Äî a Specification Freeze session (or any session that authors/corrects a `MODULE_SPEC.md`) is not exit-complete until this register reflects: (a) every unresolved `OQ-*`/`OD-*`/`FIND-*` the session's G4 reviews left open, and (b) any issue the session resolved is marked `RESOLVED` here (never deleted).
2. **[Post-flight Workflow](../workflows/postflight.md)** ‚Äî the final impact inventory (`ART-POST-001`) step includes a check of this register: newly-discovered unresolved issues are appended, resolved issues are marked `RESOLVED`, and duplicates are merged by canonical source ID before G9 is recorded.

Rules (never violated by a sync pass):

- **Append, never delete.** A resolved issue's row is updated to `status: RESOLVED` with a resolution note and revision; the row itself is retained under "Resolved Issues" (History) below.
- **Merge by canonical source ID, within a section.** Before adding a row to a given section (Global or one module), check whether its `SYNC-*`/`OQ-*`/`FIND-*`/`OD-*` ID is already aggregated by an existing row **in that same section**; if so, update that row instead of creating a duplicate. A module-section row that points back to a Global row for the same canonical source (e.g. "see `SIR-GLOB-001`") is a permitted thin cross-reference, not a duplicate ‚Äî see Schema, "Canonical source(s)".
- **Verify before restating, at section granularity.** Each module section carries its own last-verified marker (below its `## Module:` heading); a synchronization pass checks that marker first and re-reads only the module specification(s) whose marker predates their own spec's last change, rather than re-reading the whole register or a whole module section row-by-row (Constitution ¬ß17; [Context Artifacts](../constitution/CONTEXT_ARTIFACTS.md) ¬ß1.4 "verify, do not rediscover").
- **Never invent an owner.** Authority needed is recorded as `human`/etc.; owner assignment itself is reserved human authority (`SYNC-001`).
- **Archive when a section grows large.** If a module's open-row count exceeds ~25 or the "Resolved Issues (History)" table exceeds ~50 rows, split the excess into a dated archive file under `.claude/governance/archive/` (e.g. `SPECIFICATION_ISSUES_REGISTER-<module>-<year>.md`) linked from this document, rather than letting a single file grow unbounded. Not yet triggered at this register's initialization size (~8 modules, ~70 open rows, 0 resolved rows).
