# Specification Issues Register

**Artifact:** Permanent, continuously-updated cross-specification issues register (framework 1.3.0).
**Status:** LIVE — this document is never `FROZEN`; it is synchronized after every Specification Freeze session and on every platform-governance pass.
**Change class:** Project-runtime governance record. The register *mechanism* (schema, ID scheme, update protocol below) is reusable framework policy; the *entries* are project-specific (hotel-crm) evidence, consistent with Knowledge Separation (`../CLAUDE.md`).
**Scope:** Every unresolved issue surfaced by a completed Specification Freeze session (Documentation Workflow) across all module specifications in `docs/03-modules/`, plus cross-cutting/platform-level governance issues.
**Baseline revision at last full verification pass:** `ba1d78c0d7a9195394f32f6bc2206fcab7d60b8e` (`ba1d78c`, current `HEAD` at initialization).
**Canonical framework version:** see [`../VERSION.yaml`](../VERSION.yaml).

## Purpose

Before this register existed, unresolved specification issues (open decisions, blocking findings, ownership gaps, migration gaps) were discoverable only by re-reading `SYNC_STATE.yaml` and every individual `MODULE_SPEC.md`'s own "Risks, Assumptions, and Open Decisions" section. This register is the single, permanent, human-scannable index of *all* such issues, across all modules, so that:

1. A future session (or human) can see the full unresolved-issue surface without re-reading every specification.
2. Issues are never silently rediscovered, duplicated, or dropped across sessions.
3. Resolution is tracked to `RESOLVED` with history preserved — nothing is deleted.

This register **references** canonical sources (per [Context Artifacts §1.3 "Reference over copy"](../constitution/CONTEXT_ARTIFACTS.md)); it does not restate or supersede them. The canonical source for any issue is always the cited specification section or `SYNC_STATE.yaml` entry. Where this register and a canonical source disagree, the canonical source wins and the disagreement is itself logged as a new issue (Constitution §5).

## Schema

| Column | Definition |
|---|---|
| ID | Stable `SIR-<SCOPE>-<NNN>`, assigned once, never reused, never renumbered (Constitution §15) |
| Canonical source(s) | The `SYNC-*`, `OQ-*`, `FIND-*`, `OD-*`, or `ADR-*` ID(s) this entry aggregates, plus the spec path |
| Category | `ownership` \| `architecture` \| `security` \| `decision-required` \| `migration-gap` \| `consistency` \| `knowledge-sync` \| `performance` |
| Severity | `Critical` / `High` / `Medium` / `Low` / `Note`, per [Review Gates](../constitution/REVIEW_GATES.md) severity table (`decision-required` items with no live-code defect carry no severity — marked `—`) |
| Summary | One-line statement of the unresolved issue |
| Evidence | Path-and-line, revision, or decision ID |
| Authority needed | `human` / `human/architecture` / `human/product` / `human/compliance` |
| Status | `OPEN` / `BLOCKED` / `RESOLVED` |
| First recorded | Date/session this issue was first captured (in its canonical source or this register) |
| Last verified | Date this register entry was last checked against the repository |

`Unknowns` and `Limitations` are not separately tracked per entry here (that is the canonical source's job, per [Reviewer Findings](../constitution/REVIEWER_FINDINGS.md)); this register carries only status and cross-reference.

## Update Protocol (Workflow Integration)

This register is synchronized as a mandatory step of two existing workflows — no new workflow is introduced:

1. **[Documentation Workflow](../workflows/documentation.md)** — a Specification Freeze session (or any session that authors/corrects a `MODULE_SPEC.md`) is not exit-complete until this register reflects: (a) every unresolved `OQ-*`/`OD-*`/`FIND-*` the session's G4 reviews left open, and (b) any issue the session resolved is marked `RESOLVED` here (never deleted).
2. **[Post-flight Workflow](../workflows/postflight.md)** — the final impact inventory (`ART-POST-001`) step includes a check of this register: newly-discovered unresolved issues are appended, resolved issues are marked `RESOLVED`, and duplicates are merged by canonical source ID before G9 is recorded.

Rules (never violated by a sync pass):

- **Append, never delete.** A resolved issue's row is updated to `status: RESOLVED` with a resolution note and revision; the row itself is retained under "Resolved Issues" (History) below.
- **Merge by canonical source ID.** Before adding a row, check whether its `SYNC-*`/`OQ-*`/`FIND-*`/`OD-*` ID is already aggregated by an existing `SIR-*` row; if so, update that row instead of creating a duplicate.
- **Verify before restating.** A synchronization pass re-reads only the specification(s) that changed since `Last verified`; it does not re-read the whole register's evidence on every pass (Constitution §17; [Context Artifacts](../constitution/CONTEXT_ARTIFACTS.md) §1.4 "verify, do not rediscover").
- **Never invent an owner.** Authority needed is recorded as `human`/etc.; owner assignment itself is reserved human authority (`SYNC-001`).
